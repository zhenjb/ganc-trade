The off-chain side proposes a SettlementUpdate. The prover creates a real ProofBundle. The relayer submits both together as MsgSubmitBatchProof(settlementUpdate, proofBundle). x/zkdex applies the update only if proof verification succeeds.
Money flow
Deposit
User account → zkdex module account via x/bank
Claim withdrawal
zkdex module account → user/destination via x/bank
Important
x/bank is the transfer mechanism, not where money is “stored”. Custody is the zkdex module account.
State flow
Deposit tx
Creates DepositRecord{processed:false}; does not update currentStateRoot.
Submit proof
If proof is valid: update currentStateRoot, mark depositProcessed, mark nullifierUsed, create WithdrawRecord.
Claim tx
Only updates WithdrawRecord.claimed=true and transfers money. It does not change the ZK root.
Not in this MVP
No orderbook/matching in the required path. Deposit/withdraw validates the ZK settlement core. Mock data may contain one deposit and one withdrawal, but the batch schema must already support deposits[] and withdrawals[].
