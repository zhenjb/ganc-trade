Deposit flow — Cosmos SDK version
User moves Cosmos coins into x/zkdex custody. The deposit is anchored on-chain immediately, but the off-chain balance/root only becomes settled after the relayer submits a valid ZK proof batch.
User depositor
x/bank Cosmos bank module
x/zkdex settlement module
Offchain indexer/batcher
Prover ZK prover
1
User → x/zkdex: MsgDeposit(denom, amount)
No ERC20 approve step. The Cosmos transaction itself authorizes the module to move coins from the signer.
2
x/zkdex → x/bank: SendCoinsFromAccountToModule(user, ModuleName, coins)
Coins are locked in the zkdex module account / escrow.
3
x/zkdex self: write DepositRecord{deposit_id, owner, denom, amount, height, processed:false}
currentStateRoot is NOT changed yet. Deposit is on-chain, but not settled into the ZK state root.
4
x/zkdex self: emit DepositQueued(deposit_id, owner, denom, amount)
— async boundary — off-chain indexer picks up the deposit —
5
x/zkdex → Offchain: indexer reads DepositQueued event or queries DepositRecord
6
Offchain self: credit pending off-chain balance Balance[user][denom] += amount, recompute local state root
This is pending off-chain state. On-chain root has not accepted it yet.
7
Offchain self: add deposit_id to SettlementBatch.deposits[]
8
Offchain → Prover: send batch + witness
9
Prover → Offchain: return proof π
10
Relayer → x/zkdex: MsgSubmitBatchProof(settlementUpdate, proofBundle)
Settlement update and proof are submitted together in one Cosmos tx.
11
x/zkdex self: check oldStateRoot == currentStateRoot, !depositProcessed[id], then verify proof
12
x/zkdex self: if valid, set currentStateRoot = newStateRoot and depositProcessed[id] = true

Withdrawal flow — Cosmos SDK version
Withdrawal is a two-step process. Step A: a ZK batch settles the off-chain balance reduction and creates a claimable WithdrawRecord. Step B: the user calls MsgClaimWithdraw.
1
User → Offchain: create WithdrawRequest{denom, amount, destination, nonce}
2
Offchain self: check pending off-chain balance balance ≥ amount
3
Offchain self: debit off-chain balance, increment nonce, compute newStateRoot
4
Offchain self: compute nullifier = Hash(userSecret, nonce), create withdrawOutput
5
Offchain → Prover: send batch + witness
6
Prover → Relayer: return proof π + public inputs
7
Relayer → x/zkdex: MsgSubmitBatchProof(settlementUpdate, proofBundle)
8
x/zkdex self: check oldStateRoot, depositProcessed, and !nullifierUsed[nullifier]
9
x/zkdex self: verify ZK proof using public inputs
10
x/zkdex self: if valid, update root, mark nullifierUsed = true, create WithdrawRecord{claimed:false}
— user can now claim after proof batch is accepted —
11
User → x/zkdex: MsgClaimWithdraw(withdraw_id)
12
x/zkdex self: check WithdrawRecord exists and claimed == false
13
x/zkdex → x/bank: SendCoinsFromModuleToAccount(ModuleName, destination, coins)
14
x/zkdex self: set WithdrawRecord.claimed = true

ZK settlement batch — what the proof means
The relayer submits one transaction containing both the proposed settlement update and the proof. The on-chain module applies the update only if the proof is valid.
1
Offchain builds SettlementUpdate: oldStateRoot, newStateRoot, depositId, withdrawId, nullifier, withdraw amount
2
Prover creates ProofBundle{proof, publicInputs, verificationKeyId}
3
Relayer submits MsgSubmitBatchProof(settlementUpdate, proofBundle) to x/zkdex
4
x/zkdex verifies that the proof is valid for exactly this settlement update
If someone changes newStateRoot, amount, nullifier, or withdrawal output, proof verification must fail.
5
x/zkdex applies the update atomically only after successful proof verification

Invariants and edge cases
Properties that must hold in the Cosmos version, and what should happen when things go wrong.
Invariant — module account custody
Coins deposited by users must be held by the zkdex module account. Withdrawals can only be paid from that module account through x/bank.
Invariant — currentStateRoot advances only after a valid proof
currentStateRoot changes only inside MsgSubmitBatchProof, and only after proof verification succeeds. No direct admin/manual update should exist in MVP.
Invariant — withdrawal requires a WithdrawRecord
MsgClaimWithdraw never creates withdrawal rights by itself. It only pays out a previously created WithdrawRecord.
Edge case — off-chain service crashes after deposit
Safe. The deposit is already recorded on-chain as processed=false. The indexer can replay events or query pending records and rebuild the deposit queue.
Edge case — duplicate nullifier
Rejected before or during batch submission. nullifierUsed[nullifier] prevents replaying the same withdraw/state transition.
Edge case — invalid proof
The whole MsgSubmitBatchProof transaction fails. No root update, no processed deposit, no nullifier write, and no withdraw record creation.
Edge case — claim twice
Rejected by WithdrawRecord.claimed == true. The module must set claimed immediately after successful x/bank transfer.
Limitation — operator/prover availability
If the off-chain service refuses to build/prove a withdrawal, MVP may not include a forced-withdraw escape hatch. This should be documented as future work.
